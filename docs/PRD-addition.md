# Peeko — PRD Addition: RAG-Powered Lecture Materials
*Addendum to PRD v6.0 (Final) · 2026-05-01*

---

## 1. Overview & Motivation

### Problem
Peeko's summary cards are generated by calling Groq (`llama-3.3-70b-versatile`) with only two inputs: prior cards and the raw transcript window. This produces two correctness gaps:

1. **No topic grounding.** The model has no information about what the lecture is *supposed* to cover, so it cannot distinguish core concepts from tangential remarks.
2. **Generic summarization.** Without course-specific vocabulary and structure, bullets tend toward paraphrase rather than concept extraction.

### Solution
Add a Retrieval-Augmented Generation (RAG) layer. Before a session begins, users can optionally upload lecture materials (slides, handouts, syllabi). These are parsed, chunked, embedded, and stored. At every card generation and every Catch Me Up, the current transcript window is used as a similarity query to retrieve the most relevant material excerpts, which are injected into the prompt as a labeled context block.

The result: the model knows the intended lecture scope and can weight concepts accordingly. Sessions without uploaded materials are **entirely unchanged**.

### PRD §6 Update
This addition **moves "PDF lecture slides as pre-session context" from Out of Scope to In Scope** (per-session uploads only). The remaining Out of Scope items are unaffected.

## 2. User Flow

```
Dashboard
 └─ "New Session" click
     └─ /session/new  ← NEW pre-session screen
         ├─ [optional] drag-drop or file picker (PDF / DOCX / TXT, ≤ 20 MB)
         │    ├─ Uploading... → Processing... → Ready ✓  (5–15 s for typical slide deck)
         │    └─ "Start session" button enabled once ready
         ├─ "Skip & Start" button (always available, skips RAG)
         └─ → navigates to /session/:id  ← existing live session UI (unchanged)
```

- The live session view, card timeline, Catch Me Up, and PiP companion are **visually and behaviorally unchanged**.
- `has_materials = false` path adds no measurable latency to card generation (the retriever short-circuits immediately).
- Multiple file uploads per session are supported (all embedded into the same retrieval pool).

## 3. Technical Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Embeddings provider | Voyage AI `voyage-3-lite` | Anthropic's official RAG partner; leads retrieval benchmarks on dense academic content; 512-dim vectors = fast pgvector cosine queries |
| Vector store | Supabase pgvector | Reuses existing Supabase project and service-role client — no new infra |
| File storage | Supabase Storage | Native RLS, same project, free tier |
| PDF parsing | `pdf-parse` | Lightweight sync API, battle-tested in Node.js |
| DOCX parsing | `mammoth` | Standard JS DOCX → plaintext |
| Upload middleware | `multer` (memory storage, 20 MB cap) | Standard Express multipart pattern |
| Chunking strategy | ~800 chars, 100-char overlap, paragraph → sentence boundary | Balances retrieval precision vs. contextual coherence per chunk |
| Retrieval | Top-5 cosine similarity, query = current transcript window | Transcript window is already the natural query at each call site |
| Materials scope | Per-session only (no class entity) | Matches immediate requirement; cross-session reuse is deferred |

## 4. Schema Additions

Append as a new migration `backend/supabase-migration-002-rag.sql` (per CLAUDE.md: migration files are append-only; never edit a migration that has already run).

```sql
-- Enable pgvector extension
CREATE EXTENSION IF NOT EXISTS vector;

-- One row per uploaded file
CREATE TABLE lecture_materials (
  material_id    UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  session_id     UUID NOT NULL REFERENCES sessions(session_id) ON DELETE CASCADE,
  filename       TEXT NOT NULL,
  file_url       TEXT NOT NULL,          -- Supabase Storage path
  mime_type      TEXT NOT NULL,
  status         TEXT NOT NULL CHECK (status IN ('processing', 'ready', 'failed')),
  chunk_count    INTEGER DEFAULT 0,
  error_message  TEXT,
  created_at     TIMESTAMPTZ DEFAULT NOW()
);

-- One row per text chunk from a material
CREATE TABLE material_chunks (
  chunk_id     UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  material_id  UUID NOT NULL REFERENCES lecture_materials(material_id) ON DELETE CASCADE,
  session_id   UUID NOT NULL REFERENCES sessions(session_id) ON DELETE CASCADE,  -- denormalized for fast retrieval
  chunk_index  INTEGER NOT NULL,
  text         TEXT NOT NULL,
  embedding    vector(512) NOT NULL
);

-- Indexes
CREATE INDEX material_chunks_session_idx
  ON material_chunks(session_id);

CREATE INDEX material_chunks_embedding_idx
  ON material_chunks USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);

-- Fast check on the session row to skip retriever entirely when no materials
ALTER TABLE sessions ADD COLUMN has_materials BOOLEAN DEFAULT FALSE;

-- RLS: users can only access materials belonging to their own sessions
CREATE POLICY "Users can manage own lecture_materials"
  ON lecture_materials FOR ALL
  USING (session_id IN (SELECT session_id FROM sessions WHERE user_id = auth.uid()));

CREATE POLICY "Users can manage own material_chunks"
  ON material_chunks FOR ALL
  USING (session_id IN (SELECT session_id FROM sessions WHERE user_id = auth.uid()));
```

## 5. New Backend Surface

### Files to add

| File | Responsibility |
|---|---|
| `backend/src/services/materialService.ts` | Upload handler: receive file buffer → parse text (pdf-parse / mammoth) → chunk → embed → store chunks → update `lecture_materials.status` and `sessions.has_materials` |
| `backend/src/services/retrieverService.ts` | `queryRelevantChunks(sessionId, queryText, k=5): Promise<string>` — checks `has_materials`, runs cosine query via Supabase `rpc`, returns joined excerpt string (or `''` if no materials) |
| `backend/src/services/embeddingService.ts` | Thin wrapper around Voyage AI `/embeddings` endpoint; single retry on 5xx |
| `backend/src/config/voyage.ts` | Voyage base URL, API key from env, typed request/response |

### New API Endpoints

| Method | Endpoint | Request | Response |
|---|---|---|---|
| `POST` | `/session/:id/materials` | `multipart/form-data`, field `file` | `{ material_id, status: 'processing' }` |
| `GET` | `/session/:id/materials` | — | `[{ material_id, filename, status, chunk_count }]` |

Upload is fire-and-poll: the POST returns immediately with `processing` status; frontend polls `GET /materials` every 2s until `ready` or `failed`.

## 7. Prompt Changes

### `generateSummaryCard` user message (replaces current `claudeService.ts:32-40`)

```
Lecture materials (excerpts most relevant to this window):
${retrievedContext || '(none provided)'}

Previous cards (session memory):
${previousCardsSummary}

New transcript (last 5 minutes):
${transcriptWindow}

Generate the next summary card. Capture key concepts and 2-3 bullet points.
If the professor asked a question and answered it in this window, include it in the qa array.
If no Q&A detected, return qa as an empty array [].
```

### System prompt addition (appended to current `claudeService.ts:42-45`)

> *"When lecture materials are provided, prioritize concepts grounded in them; treat tangents not supported by the materials as secondary."*

### `generateCatchMeUp` user message — same pattern

```
Lecture materials (excerpts most relevant to this moment):
${retrievedContext || '(none provided)'}

Cards generated so far:
${cardsSummary}

Transcript since last card checkpoint:
${transcriptSinceCheckpoint || '(no new transcript since last card)'}

The student just snapped back to attention. Generate a recovery response: now / missed / read_first / rejoin_tip.
```

System prompt: same appended sentence as above.

## 8. Out of Scope

These items are explicitly deferred and must **not** be built as part of this addition:

- Off-topic content detection / filtering scoring
- Class entity / multi-session material reuse
- OCR for image-only PDFs (scanned slides without embedded text)
- Re-embedding if materials are updated mid-session
- Tuning the base summarization prompt beyond the one-line addition above
- Material deletion or management UI

## 9. Verification Plan

### Unit tests (write before implementation — per CLAUDE.md TDD rule)

| Test | What to assert |
|---|---|
| Chunker | Given a 3000-char input, produces chunks ≤ 850 chars each; adjacent chunks share ≥ 90 chars of overlap; no chunk is empty |
| Retriever (no materials) | When `sessions.has_materials = false`, returns `''` without calling the embedding service |
| Retriever (with materials) | Given 10 pre-seeded chunks, returns top-5 ordered by cosine similarity to the query |
| Embedding wrapper | On Voyage 500 response, retries once; on second failure throws; on 401 throws immediately |
| Material status update | After chunk insertion completes, `lecture_materials.status` is `'ready'` and `chunk_count` matches actual rows |

### Integration tests

1. Upload a known PDF → poll `GET /session/:id/materials` until `status = 'ready'` → POST a transcript chunk that contains a term from the PDF → trigger `POST /session/:id/generate-card` → assert the returned card's bullets reference at least one concept that appears in the uploaded materials.
2. Same full flow with no upload → assert card content is statistically equivalent to current baseline (no regression).
3. Fail path: upload a file with no parseable text → assert `status = 'failed'` and `error_message` is populated.

### Manual demo verification

- Side-by-side: same recorded lecture clip, run with and without slide upload. Compare card title specificity, keyword accuracy, and bullet depth.
- Confirm the `has_materials = false` path adds < 5 ms overhead to `generate-card` response time.
